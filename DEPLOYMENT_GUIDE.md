# Xeno Data Intelligence Hub - Deployment Guide

This guide covers the complete deployment of the Xeno Data Intelligence Hub across multiple cloud platforms.

---

## Deployment Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                     Frontend (Vercel)                        │
│                   Next.js 16 + TypeScript                    │
└─────────────────────┬───────────────────────────────────────┘
                      │ HTTPS API Calls
                      ▼
┌─────────────────────────────────────────────────────────────┐
│              Backend API (Render Web Service)                │
│              Litestar + Python 3.11 + PostgreSQL              │
└──────┬────────────────────────────────────────┬──────────────┘
       │                                        │
       ▼                                        ▼
┌─────────────────────┐              ┌─────────────────────┐
│  Render PostgreSQL  │              │  Render Redis       │
│   (Database)        │              │   (Queue)           │
└─────────────────────┘              └──────────┬──────────┘
                                                │
                                                │ RQ Worker
                                                ▼
                                       ┌─────────────────┐
                                       │ Railway Worker   │
                                       │ Background Jobs  │
                                       └─────────────────┘
```

## Step 1: Backend Database Setup (Render PostgreSQL)

### 1.1 Create PostgreSQL Database
1. Log in to [Render](https://render.com)
2. Navigate to Dashboard → New → PostgreSQL
3. Configure:
   - **Database Name**: `xeno`
   - **User**: `postgres` (or choose your own)
   - **Region**: Same region as other services
   - **Plan**: Free tier (for testing) or paid for production

### 1.2 Get Database Connection Details
After creation, Render will provide:
- **Internal Database URL**: `postgresql://postgres:password@host:5432/xeno`
- **External Database URL**: For local development
- **Host**: `dpg-xxxxx.oregon-postgres.render.com`
- **Password**: Auto-generated (save this securely)

---

## Step 2: Redis Queue Setup (Render Redis)

### 2.1 Create Redis Instance
1. In Render Dashboard → New → Redis
2. Configure:
   - **Name**: `xeno-redis`
   - **Region**: Same region as PostgreSQL
   - **Plan**: Free tier (for testing)

### 2.2 Get Redis Connection Details
- **Redis URL**: `redis://red-xxxxx.render.com:6379`
- **Password**: Auto-generated (save this securely)

## Step 3: Backend API Deployment (Render Web Service)

### 3.1 Update render.yaml
The `render.yaml` file is already configured. Review and update if needed:

```yaml
services:
  - type: web
    name: xeno-api
    runtime: docker
    dockerfilePath: ./backend/Dockerfile
    dockerContext: ./backend
    plan: free
    healthCheckPath: /api/docs
```

### 3.2 Connect GitHub Repository
1. In Render Dashboard → New → Web Service
2. Connect your GitHub repository
3. Select root directory
4. Build command: (handled by Dockerfile)
5. Start command: (handled by Dockerfile)

### 3.3 Configure Environment Variables
Set these in Render Environment Variables:

```bash
# System
ENV=production
DEBUG=false
SECRET_KEY=<auto-generated-by-Render>

# Database
DB_USER=postgres
DB_PASSWORD=<from-Step-1.2>
DB_HOST=<from-Step-1.2>
DB_PORT=5432
DB_NAME=xeno

# Database Pool
DB_ECHO=false
DB_POOL_SIZE=5
DB_MAX_OVERFLOW=10
DB_POOL_TIMEOUT=30

# Redis
REDIS_URL=redis://:<password>@<host>:6379/0

# AI
GROQ_API_KEY=<from-Groq-console>

# Storage
UPLOAD_DIR=./uploads
OUTPUT_DIR=./outputs
MAX_UPLOAD_BYTES=52428800
```

### 3.4 Deploy
1. Click "Deploy Web Service"
2. Render will build the Docker image and deploy
3. Monitor deployment logs for any issues
4. Once deployed, note the service URL: `https://xeno-api.onrender.com`

### 3.5 Run Database Migrations
After successful deployment, SSH into the container or use Render's shell:

```bash
# Access Render shell
cd backend
alembic upgrade head
```

## Step 4: Background Worker Deployment (Railway)

### 4.1 Create Railway Project
1. Log in to [Railway](https://railway.app)
2. Create new project
3. Deploy from GitHub repo

### 4.2 Configure Worker Service
1. Add new service → Dockerfile
2. Select `backend/Dockerfile`
3. Set command: `python -m app.workers.tasks`

### 4.3 Configure Environment Variables
Use the same database and Redis credentials from Step 3.3:

```bash
ENV=production
DEBUG=false
DB_USER=postgres
DB_PASSWORD=<same-as-Render>
DB_HOST=<same-as-Render-host>
DB_PORT=5432
DB_NAME=xeno
REDIS_URL=redis://:<password>@<host>:6379/0
GROQ_API_KEY=<same-as-Render>
UPLOAD_DIR=./uploads
OUTPUT_DIR=./outputs
```

### 4.4 Deploy Worker
1. Click "Deploy"
2. Monitor logs to ensure worker connects to Redis
3. Verify worker processes jobs from queue

---

## Step 5: Frontend Deployment (Vercel)

### 5.1 Update next.config.ts
Ensure `next.config.ts` is production-ready:

```typescript
const apiOrigin = process.env.NEXT_PUBLIC_API_URL ?? "https://xeno-api.onrender.com";
```

### 5.2 Connect GitHub Repository
1. Log in to [Vercel](https://vercel.com)
2. Add new project → Import Git Repository
3. Select root directory (or `xeno-data-hub` subdirectory)
4. Framework preset: Next.js

### 5.3 Configure Build Settings
- **Root Directory**: `./xeno-data-hub` (if using monorepo)
- **Build Command**: `npm run build`
- **Output Directory**: `.next`
- **Install Command**: `npm install`

### 5.4 Configure Environment Variables
Set in Vercel Project Settings → Environment Variables:

```bash
NEXT_PUBLIC_API_URL=https://xeno-api.onrender.com
```

### 5.5 Deploy
1. Click "Deploy"
2. Vercel will build and deploy the Next.js app
3. Once deployed, note the URL: `https://xeno-data-hub.vercel.app`

## Monitoring & Troubleshooting

### Render Monitoring
- **Backend**: Render Dashboard → Logs tab
- **PostgreSQL**: Render Dashboard → Metrics tab
- **Redis**: Render Dashboard → Metrics tab

### Railway Monitoring
- **Worker**: Railway Dashboard → Logs tab
- **Job Queue**: Check Redis connection status in logs

### Vercel Monitoring
- **Frontend**: Vercel Dashboard → Logs tab
- **Build Errors**: Vercel Dashboard → Deployments tab

### Common Issues

#### 1. Worker Not Processing Jobs
**Symptoms**: Jobs stay in "queued" status
**Solutions**:
- Check Railway worker logs for Redis connection errors
- Verify REDIS_URL environment variable
- Ensure Redis is accessible from Railway
- Check worker startup command: `python -m app.workers.tasks`

#### 2. Database Connection Errors
**Symptoms**: Backend API returns 500 errors
**Solutions**:
- Verify database credentials in environment variables
- Check Render PostgreSQL status
- Ensure database migrations ran: `alembic upgrade head`
- Check DB_HOST is correct (use internal Render URL)

#### 3. Redis Connection Timeouts
**Symptoms**: Upload fails at queue enqueue
**Solutions**:
- Verify REDIS_URL format and credentials
- Check Render Redis status
- Ensure worker and backend use same Redis instance
- Check network connectivity between services

#### 4. Frontend API Errors
**Symptoms**: Frontend shows network errors
**Solutions**:
- Verify NEXT_PUBLIC_API_URL is set correctly
- Check backend API is running and accessible
- Ensure CORS is configured (already enabled in backend)
- Check Vercel build logs for environment variable issues

#### 5. File Upload Failures
**Symptoms**: Uploads fail with file system errors
**Solutions**:
- Ensure UPLOAD_DIR and OUTPUT_DIR exist (auto-created by code)
- Check file size limits (MAX_UPLOAD_BYTES)
- Verify disk space on Render/Railway
- Check file permissions

## Scaling Considerations

### Backend API Scaling
- **Horizontal Scaling**: Add more Render web service instances
- **Database Pool**: Increase DB_POOL_SIZE and DB_MAX_OVERFLOW
- **Load Balancing**: Render automatically load balances

### Worker Scaling
- **Job Queue**: Redis can handle multiple workers
- **Horizontal Scaling**: Add more Railway worker instances
- **Worker Configuration**: Adjust RQ retry policy in code

### Database Scaling
- **Render PostgreSQL**: Upgrade to larger instance for production
- **Connection Pooling**: Tune pool settings based on load
- **Read Replicas**: Consider read replicas for read-heavy workloads

## Security Checklist

- [ ] All secrets stored in environment variables (never in code)
- [ ] DEBUG=false in production
- [ ] SECRET_KEY uses strong random value
- [ ] Database passwords are complex and rotated regularly
- [ ] API keys (Groq) are secured and rotated
- [ ] HTTPS enabled on all services
- [ ] CORS configured correctly
- [ ] Rate limiting considered for production
- [ ] File upload size limits enforced
- [ ] Input validation enabled on all endpoints
- [ ] Database migrations use proper versioning
- [ ] Logs don't contain sensitive information
- [ ] Backup strategy for PostgreSQL data

## Backup & Disaster Recovery

### Database Backups
- Render PostgreSQL offers automatic backups
- Enable point-in-time recovery for production
- Regularly export critical data for additional safety

### File Storage
- Uploads and outputs are ephemeral (can be regenerated)
- Consider object storage (S3) for persistent file storage in production
- Implement cleanup jobs for old files

### Configuration Backups
- Store environment variable templates in `.env.example` files
- Document all deployment settings
- Version control all configuration files

```bash
cd backend

# Edit .env with local settings
```

```bash
cd xeno-data-hub
npm install
```
